> 작성: 2026-06-22 | 대상: 논문용 핵심 4셀 재학습(C1/C3/C4 + C4_11s)

> 그래서 무엇을 새로 만들고 무엇을 그대로 계승했는지 기록한다. (인계·재현용)

## 0. 한 줄 요약

기존 `train.py`/`evaluate.py`는 **AIHub E01~E07 실험 전용**으로 짜여 있어 DFire 4셀 ablation에
그대로 쓸 수 없었다. 그래서 **데이터 구성 빌드 2종**과 **4셀 오케스트레이터 1종**을 신규 작성했고,
**학습 하이퍼파라미터는 `train.py`의 값을 그대로 복제**해 비교 가능성을 유지했다.

## 2. 기존 스크립트의 한계 (코드 근거)

### 2-1. `train.py` — AIHub 전용 매트릭스에 고정
- `EXPERIMENT_MATRIX`(line 31~40)가 **E01~E07(AIHub 175K) 정의로 하드코딩**.
  모델·에폭·배치만 다를 뿐, 이번 DFire 데이터 **구성 변화(C1/C3/C4)**를 표현하지 못한다.
  → 구성별 데이터셋 자체를 따로 만들어야 했다(=`build_compositions.py`).

### 2-2. `train.py` — 평가가 val 한정
- 학습 후 평가가 `val_model.val(data=data, ...)`(line 150) **한 번뿐**이고 `split` 인자가 없어
  **기본값 `val`만** 측정한다. 논문 표에는 **test 셋 수치**가 필요하다.

### 2-3. `train.py` — 클래스별 AP가 라벨 없는 배열 + val만
- `per_class_ap50`(line 156)를 저장하긴 하나 **순서만 있는 배열**이라 어느 값이 smoke인지
  문서 없이는 알 수 없고, **val에 대해서만** 저장된다.
  논문 핵심 주장("smoke AP 폭등")을 위해 **smoke/fire를 명시 라벨로, val·test 둘 다** 필요했다.

### 2-4. `train.py` — 셀 간 집계 없음
- 결과가 run별 `metrics.json`으로 흩어진다. 4셀을 **한 표(비교용)**로 모으는 기능이 없다.

### 2-5. `evaluate.py` — 단일 모델 전용
- 인자가 `--model` 하나(line 11~22)라 4셀을 한 번에 못 돌린다.
- `model.val(...)`(line 50)도 `split` 미지정 → **val만** 평가.
- `smoke_recall = metrics.box.r[0]`(line 53)로 **클래스 0=smoke를 암묵 가정** — DFire `data.yaml`의
  `names: ['smoke','fire']`와는 일치하지만, 이 가정이 코드에 주석 없이 박혀 있다.

> ※ 정정: 초기 보고에서 "train.py가 smoke AP를 안 뽑는다"고 한 것은 부정확했다.
> 실제로는 **저장은 하되 val 한정·라벨 없는 배열**이라는 것이 정확한 한계다(2-3).

## 3. 변경/신규 내용

### 3-1. `analyze_dfire.py` (신규)
DFire 각 split을 fire_only / smoke_only / both / background(빈 라벨) / missing 으로 분류,
클래스별 박스 인스턴스 수까지 출력. 구성 설계의 근거 데이터를 만든다.

측정 결과(train): fire_only 770 / smoke_only 3,836 / both 3,058 / background 6,458.

### 3-2. `build_compositions.py` (신규)
- **평가셋 고정 원칙**: val/test는 yaml에서 **원본 DFire를 그대로 참조**(중복 복사 0).
  → CLAUDE.md가 지적한 "R1 DFire / R2+ AIHub 테스트셋 불일치" 결함을 단일 DFire 벤치마크로 해소.
- **train만 구성별 생성**, 이미지는 **하드링크**(디스크 추가 0, 즉시 생성).
- 구성 정의:
  - **C1 (fire-only)**: fire 포함 이미지(fire_only+both=3,828), **smoke(0) 라벨 제거** → smoke 미학습.
  - **C3 (균형 1:1)**: both + fire_only + smoke_only(770 다운샘플) = 4,598. fire포함=smoke포함=3,828(정확히 1:1).
  - **C4 (균형+NM)**: C3 + 정상배경 전체(6,458) = 11,056.
  - 4개 yaml 생성(c1/c3/c4 + c4_yolo11s는 c4와 동일 데이터, 모델만 다름). 전 셀 `nc=2`로 통일.
- 검증: C1 smoke박스=0, C1/C3/C4 fire박스=9,638 동일 → C1→C3(균형효과)·C3→C4(NM효과) 변수 1개씩만 변화.

### 3-3. `run_4cells.py` (신규) — `train.py` 대비 추가분
| 항목 | train.py | run_4cells.py |
|------|----------|---------------|
| 학습 하이퍼파라미터 | 기준 | **동일 복제**(epochs100·imgsz640·AdamW·lr0 0.001·lrf 0.01·cos_lr·warmup3·patience30·증강) |
| 평가 split | val만 | **val + test 둘 다** |
| 클래스별 AP | 라벨 없는 배열(val) | **smoke/fire 명시 라벨**(val·test) + smoke/fire recall |
| 다중 셀 | E01~E07 고정 | **C1/C3/C4+C4_11s 순차** |
| 집계 | run별 분산 | **`runs/4cell_summary.json` 단일 표** |
| 중단/재개 | matrix만 지원 | best.pt 존재 시 셀 건너뜀 |
| 산출물 | best.pt·metrics.json | + 혼동행렬·PR곡선·results.csv(자동) + `*_val/`·`*_test/` |

## 7. 추가 변경 (2026-06-24)

### 7-1. C2 (불균형 14:1) 셀 추가
- `build_compositions.py`에 C2 추가: C1과 fire 신호 동일(fire 포함 3,828장, smoke 제거) + smoke_only 273장(=3828/14) → fire:smoke 이미지 14.0:1. `compositions/c2_imbalanced.yaml` 생성.
- `run_4cells.py` CELLS 에 `E_C2` 추가(11n, batch 64). 기존 셀은 best.pt 존재로 건너뜀.
- ⚠️ **측정 주의**: DFire 2클래스 평균 mAP에서는 C2가 C1보다 **높게** 나올 수 있음(C1 smoke AP=0이 평균을 끌어내림). "소량 smoke=노이즈" 주장은 **fire AP** 지표로 확인할 것. AIHub의 "C2<C1"은 테스트셋 불일치 산물 가능성.

### 7-2. `aihub_to_yolo.py` 클래스 인덱스 DFire 규약 정렬
- 기존 `0=fire, 1=smoke` → **`0=smoke, 1=fire`로 반전** (DFire/현 파이프라인과 통일).
- 수정 지점: `CLASS_NAMES`, `CATEGORY_MAP`(전 항목), `convert_json` 카운트, `balance_classes` 그룹화, 영상 `label_class` 배정, 관련 docstring. py_compile 검증 완료.
- 목적: 2단계 AIHub C4 보강 학습 시 per-class(smoke/fire) AP가 DFire와 같은 축에서 비교되도록.

### 7-3. `plot_dfire_ablation.py` 신규 (GPU 불요)
- `runs/4cell_summary.json` 직접 파싱 → `04_figures/dfire_4cell/fig_dfire_composition.png`(C1·C2·C3·C4, 11n 고정) + `fig_dfire_model.png`(C4 11n vs 11s) 300dpi 생성.

### 7-4. 2단계(AIHub) 선행 차단 사항 ★
- AIHub 1TB는 다운로드됐으나 **분할압축(.z01~.z08+.zip) + IRIS 임시파일(.irx*) 상태로 미해제**. 경로: `D:\AIHub_Fire\089...\3.개방데이터\1.데이터\{Training,Validation}\{01.원천데이터,02.라벨링데이터}`.
- 2단계 진입 전 필요: ① 분할압축 해제(추가 ~1TB 디스크), ② 라벨이 COCO JSON 형식인지 확인(아니면 `convert_json` 파서 수정), ③ 원천이 이미지/영상인지 확인(영상이면 `--sample_video`, 장시간).

### 7-6. AIHub 071751 압축 해제 + 스키마 분석 (2026-06-24)
- 7-Zip 설치(winget). 분할압축은 head(.zip)를 7z에 넘기면 .z01~.z0N 자동 인식.
- 압축 구성: Training 원천 TS.zip+z01~z08(**804GB**), Training 라벨 TL.zip(1.16GB), Validation 원천 VS.zip+z01(~101GB), Validation 라벨 VL.zip(0.15GB). Other/Sublabel ≈ 0B(빈 파일, 스킵).
- 해제 경로: `D:\AIHub_Fire\extracted\{Training_labels, Validation_labels, Training_source, Validation_source}`.
- **라벨 = per-image 개별 JSON 1,726,105개** (COCO 통짜 아님). 폴더: `화재 현상/이미지/{불꽃|연기}/{씬타입}/{clipid}/JSON/*.json`. 원천은 **이미지(JPG)** — 영상 아님(→ `--sample_video` 불요, 단 1.7M 프레임이라 씬 샘플링은 필요).
- **JSON 스키마**:
  - `image{width,height,filename}`, `attributes{class:"FL/SM/..", place, scene, clipname, fps}`, `categories[{1:"fl"},{2:"sm"},{3:"none"}]`, `annotations[{categories_id, bbox:[x,y,w,h]}]`
  - bbox는 COCO식 `[x,y,w,h]` → 기존 `coco_bbox_to_yolo` 재사용 가능.
  - **클래스 매핑(DFire 규약)**: categories_id `1(fl)→1(fire)`, `2(sm)→0(smoke)`, `3(none)→skip`.
- **⚠️ `convert_json`(COCO 통짜 가정)은 이 데이터에 안 맞음** → 2단계에서 **per-image JSON 파서 신규 작성** 필요. 씬 샘플링은 `attributes.scene/clipname` + 프레임 번호로 클립 단위 균등 추출 → 175K 균형(FL:SM:NM) 구성.
- 진행 상태(2026-06-24): 라벨 해제 완료, **원천(904GB) 해제 백그라운드 진행 중**.
