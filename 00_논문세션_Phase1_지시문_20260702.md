## 2. 원고 반영 지시 (섹션별)

### 2-1. 초록 (Abstract) — 헤드라인 문장 추가/교체
기존 "데이터 구성이 모델 크기보다 지배적" 유지 + 아래 한 문장 삽입(제안):
> "또한 대규모 교차 도메인 실험에서, 11배 많은 소스 데이터(114K)로 학습해도 대상 도메인(D-Fire) 성능은 회복되지 않으며(0.262), 혼합·전이학습 어느 방식도 대상 도메인 단독 학습(0.789)을 넘지 못함을 보인다 — 성능은 **대상 도메인 데이터에 종속**된다. 한편 해당 소스 데이터는 실내 화재에서 강건하다(mAP@0.5 ≈ 0.95, 클래스 통제·`[멀티시드 TBD]`)."

### 2-2. §5.6 신설 — "교차 도메인 일반화 및 데이터 의존성"
새 절을 신설하고 아래 소구조로 서술:
1. **동기**: 소스(AIHub 114K)를 대상(D-Fire)에 활용 가능한가? 도메인 갭 존재 여부.
2. **크로스도메인(A-1 + X)**: 도메인 매트릭스 표(§1) 제시 → 양방향 붕괴(0.262 / 0.400), 대각선 우세.
3. **결합·전이 실패(Design A + T1)**: 데이터 고정, 통합 전략(혼합/실외혼합/실내혼합/전이) 4종 모두 D_full(0.787) 미달 → "대상 도메인 데이터 지배" 결론. Design A로 실내/실외가 원인 아님을 규명(D-Fire test 실내≈실외).
4. **소스 내 도메인 구조(실내/실외 재평가)**: AIHub는 자기 도메인 내에서 실내(0.946/균형0.958) ≫ 실외(0.824/0.821), 클래스 통제 후에도 Δ+0.137 견고 → 실질 도메인 난이도 차이. 건축물 비화재보 동기와 연결.
5. **함의**: 규모·전이보다 대상 도메인 수집이 우선. 실내 배포에는 AIHub 소스가 유효.

### 2-3. §5.4 / 결과표 — D_full 헤드라인 반영
- 절대성능 헤드라인을 **D_full 11m 0.789**로(기존 ablation C4 0.736과 구분해 별도 "최종 모델"로 명시 — C5 아님).
- prior work 비교(CCi 78.5%, YOLOGX 80.9%)와 대등/근접 서술. 단, **평가셋 상이 주의**(우리 D-Fire test) 각주.

### 2-4. 결론 (Conclusion) — 헤드라인 교체
제안:
> "본 연구는 (1) 데이터 구성(균형·정상배경)이 모델 크기보다 지배적이며, (2) 성능은 대상 도메인 데이터에 종속되어 규모·혼합·전이로 대체 불가함을 보였다. 실내 화재 감시라는 응용 맥락에서는 소스 도메인(AIHub)이 강건한 성능(≈0.95)을 제공한다."

## 3. 주의 / 금지 (SSOT·컨벤션)
- **R1(0.786)과 R2+ 를 같은 비교열에 넣지 말 것**(평가셋 상이 — DFire vs AIHub test).
- 통계 검증 컨벤션: **YOLO11n 기준 n=3 mean±std** (D-Fire E11~E20과 동일). AIHub 멀티시드도 11n으로 통일.
- 11s(0.918)/11m(0.914) 단일시드는 "모델 용량 경향 참고용"으로만(부록/보조), 헤드라인 아님.
- 그림: 결과 그림(혼동행렬·PR곡선·정성)은 가중치 삭제로 재생성 불가 항목 존재 — TRAINING_LOG의 막대그래프 계열은 수치로 재현 가능. 도메인 매트릭스는 표/막대로 신규 작성 가능.
