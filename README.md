## Git 워크플로우 (중요 — 브랜치 나누지 않음)

- **브랜치는 `yhh` 하나만 사용합니다.** 각자 별도 브랜치를 파지 않습니다 — 이 프로젝트는 인원이 적고
  `TRAINING_LOG.md`라는 단일 SSOT 파일 중심으로 굴러가서, 브랜치를 나눴다가 나중에 합치는 것보다
  그때그때 최신 상태 위에 이어 쓰는 편이 충돌이 적습니다.
- **push 전에 항상 실행**:
  ```
  git pull --rebase origin yhh
  ```
- **`TRAINING_LOG.md`에 자기 실험을 추가할 때는 반드시 자신만의 섹션 제목으로 구분해서 append**하세요
  (예: `## [본인 이니셜] — 실험명 (날짜)`). 같은 파일이라도 섹션이 겹치지 않으면 충돌이 거의 안 납니다.
- 가중치(`runs/`, `*.pt`)는 `.gitignore` 대상이라 git에 올라가지 않습니다 — 로컬에 보존하고, 필요한
  수치·결론만 `TRAINING_LOG.md`에 텍스트로 기록하세요.

## 데이터·재현 관련 핵심 주의사항

- **평가 기준(test)은 D-Fire 공식 Kaggle 분할(`smoke-fire-detection-yolo`)의 test 4,306장으로 고정.**
  ablation(C1~C4) 전 구성에서 val/test는 절대 바꾸지 않고 train 안에서만 구성을 바꿈.
- **⚠️ 알려진 데이터셋 이슈**: 이 Kaggle 분할은 train↔test 간 근접/완전 중복이 존재함(약 22%가 픽셀
  동일 수준, 2026-07-01 확인). ablation의 상대 비교(C1→C4 등)에는 영향 없으나(전 셀에 동일하게 적용됨),
  절대 성능 수치는 낙관적일 수 있음. 이 문제로 새 실험을 설계할 때는 먼저 TRAINING_LOG에서 관련 절을
  검색해 중복 여부를 확인할 것.
- **클래스 규약**: 0=smoke, 1=fire로 전 실험 통일(D-Fire, AIHub 공통).
- **AIHub(071751) 데이터**는 D-Fire와 별개 벤치마크(도메인 차이 확인됨 — AIHub 학습 모델을 D-Fire test로
  평가하면 성능이 크게 떨어짐). 두 데이터셋 수치를 직접 비교하지 말 것.

## 재학습 시 스크립트 사용 순서

1. `aihub71751_to_yolo.py` 또는 D-Fire 원본 → YOLO 형식 변환/구성 (`build_compositions.py`)
2. `train.py` / `train_matrix.py` / `run_*.py` — YOLO11 파인튜닝
3. `evaluate.py` — 평가
4. `export_trt.py` — Jetson TensorRT FP16 변환·FPS 측정

세부 하이퍼파라미터(에폭, patience, 옵티마이저 등)와 각 실험의 정확한 설정은 `02_data_ssot/TRAINING_LOG.md`의
해당 실험 섹션에 전부 기록되어 있음 — 재현 전에 먼저 확인할 것.
